Focusable Elements (Query Strict) - ally.js Compatibility Table

The following tables show the differences between what browsers consider focusable and what ally.js identifies using strategy: strict.

Note that touch devices (without a physical keyboard) only show elements as tabbable (keyboard focusable), that can be navigated to through the on-screen keyboard (or "virtual keyboard").

Footnotes

  • The DOM Element Interface is missing the focus method.
  • The focus event is not emitted upon the element becoming the active element (document.activeElement).
  • The :focus CSS pseudo class is not set on the element when it is the active element (document.activeElement).
  • The element becomes the active element (document.activeElement) when an element of contentDocument or the shadowRoot has focus.

  • Note 1

    Considering the <html> and <body> element focusable should have no benefit to a website (unless explicitly made focusable by adding the tabindex attribute), which is why ally.js considers the <html> and <body> element inert.

  • Note 2

    The <html> element itself is not considered focusable, but some browsers may give it focus when focus is passed from browser UI to the document.

  • Note 3

    The <body> element itself is not considered focusable, but it is has focus (i.e. is document.activeElement) if no other element has focus.

  • Note 6

    The activation behavior of the <label> element is not defined beyond »… should match the platform's label behavior.«

  • Note 7

    Internet Explorer redirects focus from <label> to the referenced form control element upon mouse click, but not on script focus via element.focus().

  • Note 21

    Technically the content of an iframe can be accessible to script.

  • Note 22

    Because of the Same Origin Policy we can't reliably look inside <iframe>s to determine their content, which is why ally.js considers all <iframe> content inert

  • Note 23

    There is no API to interact with the content document of an <embed> element

  • Note 24

    Because there is no API to dive into the content document of an <embed> element, ally.js considers all <embed> content inert.

  • Note 26

    Technically the content of an <object> element can be accessible to script.

  • Note 27

    Because of the Same Origin Policy we can't reliably look inside <object>s to determine their content, which is why ally.js considers all <object> content inert.

  • Note 32

    The behavior of the <object> element depends on the content it presents. The most common content types used with <object> are SVG and SWF.

  • Note 33

    This element is actually tabbable (keyboard focusable).

    But when tabbing to it, the <kbd>Tab</kbd> behavior for the entire document breaks, as focus remains stuck on the browser UI.

    This might be related to Trident 1109008.

  • Note 34

    SVG 1.1 does not specify much in respect to accessibility. SVG 2 will bring the tabindex attribute. SVG Tiny 1.2 knows the focusable attribute.

  • Note 35

    By registering a focus event listener the element becomes focusable. Blink 445798, WebKit 140024.

    This is undetectable because elements don't provide a list of their registered event handlers.

  • Note 36

    Elements made focusable by way of adding a focus event listener cannot be identified by ally.js

  • Note 39

    The <details> element is specified in HTML 5.1, but not implemented everywhere.

  • Note 40

    This element should be keyboard focusable (tabbable), but is currently only focusable because of WebKit 151767

  • Note 41

    This element should not be focusable (tabbable), but is currently only focusable because of WebKit 151767

  • Note 46

    Firefox may hide elements from the document's tabbing sequence if they're enclosed by two images referencing the same image map - Gecko 1116126.

  • Note 47

    This test is not about an element's focusable state, but about potentially content that's potentially hidden from the tabbing sequence.

  • Note 48

    Firefox transforms <keygen> to <select _moz-type="-mozilla-keygen"> while parsing HTML.

  • Note 49

    The <keygen> element is poorly supported, practically never used and has seen intent to deprecate

  • Note 50

    Because the <keygen> element is poorly supported, ally considers all <keygen> elements focus-relevant but not focusable.

  • Note 54

    Focus is redirected to the labeled control.

  • Note 55

    Focus is redirected to the nested labeled control.

  • Note 81

    When this element is the activeElement, the reference element <object usemap="…"> has the following state:

    • is not the activeElement in its context
    • does not have the :focus CSS pseudo class applied
    • does not receive a focus event

  • Note 82

    When this element is the activeElement, the reference element <object usemap="…"> has the following state:

    • is not the activeElement in its context
    • does not have the :focus CSS pseudo class applied
    • receives a focus event
